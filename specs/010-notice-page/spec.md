# 기능 명세서: 공지사항 페이지

## 개요

사용자가 서비스 공지사항 목록을 보고, 검색하며, 개별 공지사항 상세 내용을 읽을 수 있는 기능입니다. 마이페이지에서 공지사항으로 이동할 수 있으며, 모바일 환경에 맞게 목록은 처음 일부만 로드한 뒤 "더보기"로 추가 로드하는 방식으로 제공됩니다.

### 배경

서비스 운영 공지·업데이트 내용을 사용자에게 전달할 채널이 필요합니다. 사용자가 마이페이지에서 고객 센터 위의 진입점을 통해 공지사항 목록에 접근하고, 제목·본문 검색과 상세 보기를 통해 원하는 정보를 빠르게 찾을 수 있어야 합니다.

### 목적

- 사용자가 공지사항 목록을 한눈에 보고, 필요한 공지만 골라 읽을 수 있도록 함
- 제목·본문 기준 검색으로 관심 공지를 빠르게 찾을 수 있도록 함
- 중요 공지(고정 공지)가 목록 상단에서 시각적으로 구분되어 보이도록 함
- 모바일에서도 스크롤 흐름을 유지하면서 의도치 않은 끝까지 스크롤을 줄이고, 하단 영역을 보호하기 위해 "더보기" 방식으로 추가 목록을 로드함

## 기능 요구사항

### FR-1: 공지사항 목록 조회

**설명**: 사용자가 공지사항 목록 화면에 진입하면 최근 공지사항 목록이 표시됩니다.

**수락 기준**:
- 공지사항 페이지 진입 시 자동으로 목록을 조회하여 표시함
- 한 번에 10개 단위로 목록을 가져옴
- 처음에는 10~15개 정도의 목록만 표시함
- 목록에는 각 공지의 제목, 게시 시점(상대 시간 예: "5일 전", "7일 전")이 표시됨
- 고정(공지) 여부가 있는 항목은 "공지" 라벨 등으로 시각적으로 구분됨
- 전체 공지사항 개수(예: "총 22개의 공지사항")가 사용자에게 표시됨
- 목록 조회 중에는 로딩 상태를 표시함
- 목록 조회 실패 시 사용자에게 오류 메시지를 표시함
- 목록이 비어 있는 경우 적절한 안내 메시지를 표시함

### FR-2: 더보기(추가 로드)

**설명**: 사용자가 목록 하단의 "더보기" 버튼을 누르면 다음 10개 단위 데이터가 기존 목록 아래에 이어서 표시됩니다.

**수락 기준**:
- 목록 하단에 "더보기" 버튼이 노출됨
- 더 불러올 데이터가 있을 때만 "더보기" 버튼이 보이거나 활성화됨
- "더보기" 클릭 시 다음 10개 단위 데이터를 요청하여 기존 목록 아래에 덧붙여 표시함
- 추가 로드 중에는 로딩 상태를 표시함
- 추가 로드 실패 시 사용자에게 오류 메시지를 표시하고, 필요 시 재시도 방법을 제공함
- 더 불러올 데이터가 없으면 "더보기" 버튼을 숨기거나 비활성화함

### FR-3: 공지사항 검색

**설명**: 사용자가 검색어를 입력하면 제목·본문 기준으로 공지사항을 검색한 결과만 목록에 표시됩니다.

**수락 기준**:
- 검색 입력 필드가 제공되며, 플레이스홀더 등으로 "공지사항 검색" 의도가 드러남
- 사용자가 검색어를 입력하고 검색을 실행하면(엔터 또는 검색 버튼 등) 제목·본문 기준 검색 결과만 목록에 표시됨
- 검색 결과가 없으면 "검색 결과가 없습니다" 등 안내 메시지를 표시함
- 검색 시에도 전체 개수 또는 검색 결과 개수가 사용자에게 표시됨
- 검색 후 목록도 10개 단위 로드 및 "더보기"로 추가 로드 가능함

### FR-4: 공지사항 상세 조회

**설명**: 사용자가 목록에서 특정 공지 항목을 선택하면 해당 공지의 상세 내용을 볼 수 있는 화면으로 이동합니다.

**수락 기준**:
- 목록의 각 항목은 상세로 이동할 수 있음(클릭·터치 가능 영역이 명확함)
- 항목 선택 시 해당 공지의 상세 화면으로 이동함
- 상세 화면에서는 해당 공지의 제목, 게시 일시, 본문 내용 등이 표시됨
- 상세 조회 중에는 로딩 상태를 표시함
- 상세 조회 실패(존재하지 않거나 오류) 시 사용자에게 오류 메시지를 표시하고, 목록으로 돌아갈 수 있는 방법을 제공함

### FR-5: 진입 경로

**설명**: 사용자는 마이페이지에서 "고객 센터" 메뉴 위에 있는 공지사항 진입 메뉴를 통해 공지사항 목록 페이지로 이동할 수 있습니다.

**수락 기준**:
- 마이페이지 내비게이션 영역에 공지사항으로 이동하는 메뉴가 있으며, 고객 센터 메뉴 위에 위치함
- 해당 메뉴 선택 시 공지사항 목록 페이지로 이동함

### FR-6: 헤더 및 공통 UI

**설명**: 공지사항 목록·상세 페이지의 헤더는 앱에서 사용하는 공통 헤더 컴포넌트를 사용하며, 페이지 닫기 등 일관된 동작을 제공합니다.

**수락 기준**:
- 공지사항 목록·상세 화면의 상단 헤더는 프로젝트 공통 헤더 컴포넌트를 사용함
- 헤더를 통해 페이지 닫기(이전 화면 또는 지정된 대상으로 이동)가 가능함

## 사용자 시나리오

### 시나리오 1: 공지사항 목록 보기

1. 사용자가 마이페이지에서 공지사항 메뉴를 선택함
2. 공지사항 목록 페이지가 열리고, 최근 공지 10~15개 정도가 표시됨
3. 사용자는 "총 N개의 공지사항" 문구로 전체 개수를 확인함
4. 각 항목에서 제목, 게시 시점(상대 시간), 고정 공지 여부("공지" 라벨)를 확인함
5. 더 보고 싶으면 목록 하단 "더보기"를 눌러 다음 10개가 이어서 표시됨

### 시나리오 2: 공지 검색하기

1. 사용자가 공지사항 목록 페이지의 검색 필드에 키워드를 입력함
2. 검색을 실행함(엔터 또는 검색 버튼)
3. 제목·본문에 해당 키워드가 포함된 공지만 목록에 표시됨
4. 검색 결과가 없으면 "검색 결과가 없습니다" 안내를 봄
5. 검색 결과가 많으면 "더보기"로 추가 결과를 불러올 수 있음

### 시나리오 3: 공지 상세 읽기

1. 사용자가 공지사항 목록에서 관심 있는 항목을 선택함
2. 해당 공지의 상세 화면으로 이동함
3. 제목, 게시 일시, 본문 내용을 읽음
4. 헤더의 닫기 등으로 목록 또는 이전 화면으로 돌아갈 수 있음

### 시나리오 4: 목록이 비어 있는 경우

1. 사용자가 공지사항 목록(또는 검색 결과) 페이지에 진입함
2. 표시할 공지가 없음
3. "공지사항이 없습니다" 또는 "검색 결과가 없습니다" 등 상황에 맞는 안내 메시지가 표시됨

### 시나리오 5: 추가 로드 실패

1. 사용자가 "더보기"를 눌러 다음 목록을 요청함
2. 네트워크 또는 서버 오류로 추가 로드에 실패함
3. 사용자에게 오류 메시지가 표시되고, 재시도 방법(버튼 등)이 제공됨

## 성공 기준

### SC-1: 목록 조회

- 공지사항 목록 페이지 진입 후 3초 이내에 첫 목록이 표시됨
- 표시된 목록이 서버 데이터와 일치함(제목, 날짜, 고정 여부 등)

### SC-2: 더보기 경험

- "더보기" 클릭 후 3초 이내에 추가 목록이 기존 목록 아래에 이어서 표시됨
- 추가 로드 후에도 기존에 보던 스크롤 위치가 유지되거나, 새로 추가된 영역으로 자연스럽게 이어짐

### SC-3: 검색

- 검색 실행 후 3초 이내에 검색 결과 목록이 표시됨
- 검색 결과가 제목·본문 기준으로 검색어와 일치함

### SC-4: 상세 이동

- 목록에서 항목 선택 시 2초 이내에 상세 화면이 표시됨
- 상세 화면에 해당 공지의 제목·일시·본문이 정확히 표시됨

### SC-5: 사용자 만족

- 사용자가 목록 조회, 검색, 더보기, 상세 보기를 통해 원하는 공지 정보를 얻을 수 있음
- 오류 발생 시 메시지와 복구 방법이 명확히 제공됨

## 핵심 엔티티

### 공지사항 목록 항목 (Notice List Item)

- **id**: 공지사항 고유 식별자
- **title**: 제목
- **imageUrl**: 대표 이미지 URL (없으면 null)
- **isPinned**: 고정(공지) 여부 — 목록에서 "공지" 라벨 표시에 사용
- **isVisible**: 노출 여부
- **createdAt**: 게시 일시 — 목록에서 상대 시간("N일 전") 또는 절대 시간 표시에 사용

### 공지사항 상세 (Notice Detail)

- **id**: 공지사항 고유 식별자
- **title**: 제목
- **content**: 본문 내용
- **imageUrl**: 대표 이미지 URL (없으면 null)
- **isPinned**: 고정(공지) 여부
- **isVisible**: 노출 여부
- **createdAt**: 게시 일시
- **updatedAt**: 수정 일시

### 공지사항 목록 응답 (List API 응답)

- **success**: 요청 성공 여부
- **data**:
  - **items**: 공지사항 목록 배열 (목록 항목 구조)
  - **total**: 전체 공지사항 개수(또는 검색 시 검색 결과 개수)
  - **limit**: 한 번에 요청한 개수
  - **offset**: 건너뛴 개수
- 다음 데이터 존재 여부는 total과 현재 로드된 개수로 "더보기" 노출 여부 판단에 사용

### 공지사항 상세 응답 (Detail API 응답)

- **success**: 요청 성공 여부
- **data**: 공지사항 상세 객체 (상세 구조)

## 비기능 요구사항

### NFR-1: 성능

- 목록 최초 로드: 3초 이내
- 더보기 추가 로드: 3초 이내
- 검색 결과 표시: 3초 이내
- 상세 화면 표시: 2초 이내

### NFR-2: 사용자 경험

- 목록·검색·더보기·상세 조회 시 로딩 상태가 표시됨
- 오류 시 사용자 친화적 메시지와 복구(재시도, 이전 화면) 방법이 제공됨
- 모바일에서 터치 영역이 충분히 넓어 잘못된 클릭이 최소화됨

### NFR-3: 접근성

- 키보드·스크린 리더로 목록 탐색, 검색, 항목 선택, 더보기, 닫기가 가능함
- 버튼·링크에 적절한 라벨 또는 aria 속성이 제공됨

## 제약사항

- 목록은 10개 단위로 요청하며, 처음에는 10~15개만 표시하고 이후 "더보기"로 10개씩 추가 로드함
- 검색은 제목·본문 기준으로만 수행함(서버 동작에 따름)
- 헤더는 공통 헤더 컴포넌트를 사용하며, 별도 전용 헤더를 두지 않음

## 가정사항

- 사용자는 로그인된 상태로 마이페이지에 접근함
- 공지사항 목록·상세·검색을 제공하는 백엔드가 존재하며, 목록은 limit·offset(또는 이에 상응하는 방식), 검색은 키워드 파라미터로 동작함
- 고정 공지 여부를 나타내는 필드가 응답에 포함되어, "공지" 라벨 표시에 사용할 수 있음
- 게시 일시가 응답에 포함되어, 상대 시간("N일 전") 또는 절대 시간 표시가 가능함

## 엣지 케이스

### EC-1: 목록이 비어 있음

- 공지가 하나도 없거나 검색 결과가 없을 때, 빈 목록 대신 "공지사항이 없습니다" / "검색 결과가 없습니다" 안내 메시지를 표시함

### EC-2: 네트워크·서버 오류

- 목록·검색·더보기·상세 조회 실패 시 오류 메시지를 표시하고, 재시도 또는 이전 화면으로 돌아가기 옵션을 제공함

### EC-3: 상세 공지가 삭제되었거나 존재하지 않음

- 이미 삭제되었거나 잘못된 식별자로 상세 조회 시 "공지를 찾을 수 없습니다" 등 메시지를 표시하고, 목록으로 돌아갈 수 있게 함

### EC-4: 검색어 없이 검색 실행

- 검색어가 비어 있을 때 검색 실행 시, 전체 목록을 다시 보여주거나 "검색어를 입력해 주세요" 안내 중 하나로 일관되게 처리함

### EC-5: 더 불러올 데이터가 없음

- 다음 10개가 없을 때는 "더보기" 버튼을 숨기거나 비활성화하여, 불필요한 요청과 혼란을 방지함
